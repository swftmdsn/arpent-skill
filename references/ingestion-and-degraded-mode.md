# Ingestion and degraded mode

Arpent is designed to operate **with** a CLI. But there are situations where the CLI is unavailable or not yet installed:

- The user has the skill but hasn't installed the Python CLI
- The user is operating Arpent through a different harness without CLI access
- A specific commands hasn't been implemented yet (Phase 1 stubs)

In these cases, the agent operates in **degraded mode**: it writes ordinary markdown directly and follows all routing and frontmatter rules manually. It does not emulate coordinated database or multi-file operations.

## When to use degraded mode

Detect degraded mode by:

1. Checking if the `arpent` command exists in the environment (`which arpent` or attempting `arpent --version`)
2. The user explicitly says they want to write files manually

If any of these conditions hold, operate in degraded mode for the entire session.

## Degraded-mode rules

### What the agent CAN do without CLI

- Write markdown files directly with full universal frontmatter
- Place files at correct paths via routing rules in `routing.md`
- Read and search vault files using filesystem tools (`grep`, `find`, file viewer)
- Append to fleeting daily files
- Move files between directories (with announce-and-confirm)
- Create folders for new projects/areas manually only when the CLI is unavailable

### What the agent CANNOT do without CLI

- Operate todo dual state, linear dissolution, sweep, or other coordinated state
- Run sweep operations
- Maintain `index.json` and `sidecar.json` consistency at scale
- Run cron jobs

External memory is not part of Arpent degraded mode. Use the host's memory
interface when available; otherwise state that the fact or trait was not
persisted.

### No manual deferred-memory queue

`06_indexes/pending_db_writes.yaml` is owned by the implemented
`arpent session end` workflow. Arpent has no `pending apply` command and agents
must not append to or flush this file manually. It records deferred intent for
future integration work; it does not prove that external memory was updated.

### Manual sidecar updates

`sidecar.json`, `index.json`, `context_index.json`, and the search database are normally generated by `arpent index`. In degraded mode:

- Don't try to maintain them - they will be out of sync
- After several manual file changes, advise the user: "Run `arpent index` later to rebuild the indexes."

This is acceptable - the indexes are derivatives, the markdown files are canonical.

## Manual file creation - full procedure

Step-by-step for creating a new note in degraded mode:

### 1. Determine the type and routing

Use `memory-layers.md` to confirm vault is the right layer. Use `routing.md` to determine the destination.

### 2. Generate the ID

Format: `<type>-<YYYYMMDD>-<letter>`

The `<letter>` is `a` for the first note of that type on that day, `b` for the second, etc.

To find the next letter, scan frontmatter IDs matching `<type>-<YYYYMMDD>-*` in the entire vault and pick the next available letter. IDs are not filenames.

```bash
rg '^id: <type>-<YYYYMMDD>-' --glob '*.md'
```

### 3. Build the frontmatter

Use the schema from `frontmatter.md`. Apply the policies:

- Identity and system fields: normalize `title` to lowercase ASCII `snake_case`; generate `id`; fill `created` and `modified`
- Routing-critical: fill based on user intent
- Subjective fields: `appreciated: null`, `importance: null`, `pinned: false`
- Provenance: `source` and `link` per coherence rules
- `author: agent` if creating without explicit user request, otherwise `user`

### 4. Validate the frontmatter

Before writing, check:

- `project` and `resource` are not both set; `area` may accompany either as context
- `source` and `link` are coherent (per cross-table)
- `appreciated` and `importance` are null (no agent-filled values)
- `id` is unique in the vault
- `type` is in the enum
- `depth`, when meaningful, is an integer from 1 to 5

If any check fails, fix or surface the issue.

### 5. Compute the path

Apply the pure routing function. If ambiguous → `00_inbox/unsure/` with `_reason.txt`.

### 6. Announce

Show the user:
- The full proposed frontmatter
- The destination path
- Wait for confirmation

### 7. Write

Use the file-writing tool. The filename is `<title>.md`, with the same normalized value as frontmatter `title`. Write the frontmatter followed by `---`, then a simple Markdown body with no repeated H1 or source URL. Extracted knowledge must be autonomous from its source framing.

### 8. Confirm

Report what was created with the structured format from the main SKILL.md.

## Raw inbox ingestion when the CLI is available

Do not use degraded direct writes for raw inbox files while the CLI works. First
run `arpent triage --json`, then preview the explicit transaction:

```bash
arpent note ingest 00_inbox/source.txt --title "Source" --dry-run --json
arpent note ingest 00_inbox/source.txt --title "Source" \
  --source-hash <sha256-from-preview>
```

UTF-8 text becomes the full body of a new complete structured note. Malformed
frontmatter is preserved as body text with a warning. A binary/non-UTF-8 file
remains byte-for-byte untouched and cannot contain YAML. With `--attachment`
and a selected project, area, or resource, the transaction moves that original
to the home's `attachments/` and creates a separate Markdown companion reference
note with complete universal frontmatter whose `link` points to the attachment.
Without a final home, the original stays in inbox and the companion reference
note is untriaged. The transaction never overwrites note or attachment
collisions and removes the inbox source only after the destination commits.

For a batch, preview every item, obtain one confirmation, then invoke the
individual transactions sequentially. Report applied, skipped, and failed items
separately because the batch as a whole is not atomic.

## External tree import when the CLI is available

`note ingest` is for one source already under `00_inbox/`. Do not manually copy a
large external hierarchy into the inbox. Use the dedicated reviewed migration
pipeline instead:

```bash
arpent import scan ~/legacy --output ~/migration/legacy-plan.json
arpent import review ~/migration/legacy-plan.json
arpent import validate ~/migration/legacy-plan.json --sources
arpent import apply ~/migration/legacy-plan.json --dry-run --json
arpent import apply ~/migration/legacy-plan.json
```

The scan is read-only, folder decisions are inherited, and actual application
copies each source through the existing ingestion guarantees. Import state is
resumable and batches may partially succeed. External-tree import is not safely
emulated in degraded mode; preserve the source and wait for the CLI.

## Batch ingestion (without CLI)

For ingesting multiple items at once (e.g., the user dumps a list of fleeting thoughts), batch them but still announce:

```
About to create 5 fleeting notes in 00_inbox/fleeting/2026-04-19.md:
  09:42 - "thought 1"
  10:15 - "thought 2"
  14:30 - "thought 3"
  ...

Proceed? [y/n]
```

On `y`, append all 5 in one operation. Don't ask 5 separate times.

## Reading the vault without CLI

Common operations and their non-CLI equivalents:

| Operation | Without CLI |
|---|---|
| `arpent status` | List files in `00_inbox/`, count maturing/active/etc by grep frontmatter |
| `arpent search "x"` | Search live vault files recursively |
| `arpent note read <id>` | Find file with matching id, display content |
| `arpent triage` | Walk inbox files manually, propose routing for each |
| `arpent project create <name>` | Manually create project context and `notes/`, `drafts/`, `attachments/` only after confirmation |
| `arpent note ingest <path>` | Preserve raw source and create the equivalent complete note/attachment transaction manually only when safe |
| `arpent archive <target>` | Move file to `04_archives/<YYYY_qX>/...` and update frontmatter `status: archived` |
| `arpent efforts` | find notes and contexts with `status: active`, then group explicit `effort_cadence` + `effort_level`; missing profiles are `unclassified` |
| `arpent session end` | Append a `_context.md` session block by default; write `MEMORY.md` only after explicit user opt-in; do not emulate the CLI-owned queue |

## End-of-session protocol in degraded mode

The local protocol is partially doable without the CLI:

1. **Update `_context.md`** with an announced, confirmed direct edit, preserving body sections and the complete universal frontmatter set.
2. **Do not create or update `MEMORY.md` by default.** Only do so when the user explicitly opts into that log for this close; do not read it later without separate user opt-in.
3. **Persist observations or traits through the host interface**, or report that they were not persisted.

Do not create a local substitute for unavailable external memory.

## Limits of degraded mode

Some operations are **not** safely doable without CLI:

- **Atomic multi-file operations** (e.g., dissolving a linear note: needs to update source + create children + maintain `parent`/`extracted_to` consistency)
- **DB consistency** (FTS5 indexes get out of sync if updated outside the CLI)
- **Sweep operations** (require reading `tools.yaml`, walking many files, updating statuses)
- **Cron** (requires the CLI tick mechanism)

For these, defer or wait for CLI availability.

If the user insists on doing complex operations without CLI, warn explicitly:

> "Dissolving a linear note in degraded mode is risky - it requires atomic updates across the source and all children. I recommend installing the CLI first. If you want to proceed manually, I'll do it step by step but you'll need to verify consistency afterward."

## Transition from degraded to CLI operation

When the user installs the CLI, run reconciliation:

1. `arpent index` - rebuild inventory, sidecar, context, and search indexes from the filesystem; this never invokes AI
2. `arpent status` - verify health

After this, normal CLI operation resumes in the vault's existing full or minimal mode.

## Why this matters

Degraded mode is **not** a fallback or a hack. It is a designed property of the system:

- Markdown files remain the source of truth
- DBs are derivatives that can be rebuilt
- The CLI is a productivity layer, not a dependency for survival

A user who clones the vault on a fresh machine without installing anything can still read all their notes with `cat`, search with `grep`, and edit with any text editor. This is the file-over-app principle in practice.
